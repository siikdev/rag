# Stage 0 — Naive RAG: 구조와 한계

## RAG란?

**Retrieval-Augmented Generation (RAG)**는 LLM의 두 가지 근본적인 문제를 해결합니다.

| 문제 | 원인 | RAG의 해결 방법 |
|------|------|----------------|
| **Hallucination** | 모델이 훈련 데이터에 없는 내용을 지어냄 | 실제 문서를 컨텍스트로 제공 |
| **Knowledge Cutoff** | 훈련 이후 정보를 모름 | 최신 문서를 실시간 검색 |

---

## Naive RAG 흐름

```
[Indexing 단계 — 사전 작업]
문서 → 청킹 → 임베딩 → Vector DB 저장

[Query 단계 — 실시간]
질문 → 임베딩 → Vector DB 유사도 검색 → 상위 k개 컨텍스트 → LLM → 답변
```

### 각 단계 설명

**청킹 (Chunking):**
LLM의 컨텍스트 길이 제한 때문에 문서를 작은 단위로 잘라야 합니다.
너무 크면 컨텍스트 초과, 너무 작으면 의미가 단절됩니다.

**임베딩 (Embedding):**
텍스트를 벡터(숫자 배열)로 변환합니다.
의미가 비슷한 텍스트는 벡터 공간에서 가깝게 위치합니다.

**유사도 검색:**
질문 벡터와 가장 가까운 문서 벡터를 찾습니다.
주로 **Cosine Similarity**를 사용합니다.

## Naive RAG의 한계

Stage 0에서 베이스라인을 만드는 이유는, 이 한계를 직접 확인하기 위함입니다.

| 한계 | 문제 상황 | 해결 Stage |
|------|-----------|-----------|
| **키워드 미스매치** | "심근경색" 검색 시 "heart attack" 문서를 못 찾음 | Stage 1 (Sparse) |
| **쿼리 표현의 한계** | 질문이 너무 짧거나 모호하면 관련 문서를 못 찾음 | Stage 1 (HyDE, RAG-Fusion) |
| **순위 부정확** | 유사도 점수가 높아도 실제로 관련 없는 문서가 섞임 | Stage 2 (Reranking) |
| **글로벌 이해 부족** | "이 논문의 전체 주장은?" 같은 질문에 약함 | Stage 5 (Graph RAG) |
